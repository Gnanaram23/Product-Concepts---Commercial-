Build a full-stack web application called ClearAuth — a Prior Authorization (PA) 
and Re-Authorization (RA) platform for healthcare. Stack: React + TypeScript 
frontend, Express.js backend, in-memory data store. Use Vite for bundling. 
Run frontend on port 5173, backend on port 3001. Start command: npm run dev 
(concurrently runs both).

// src/types/index.ts

export type PAStatus = 'pending' | 'review' | 'approved' | 'denied';
export type Priority = 'normal' | 'high';
export type SubmissionChannel = 'fhir' | 'portal' | 'fax';
export type UserRole = 'provider' | 'patient' | 'payer';
export type UrgencyLevel = 'normal' | 'urgent';

export interface PACase {
  id: string;                   // "PA-249761"
  patientId: string;
  patientName: string;
  patientDob: string;           // "11/05/1975"
  drug: string;                 // "Ozempic"
  ndcCode?: string;
  condition: string;            // "Rheumatoid arthritis, unspecified"
  icd10Code: string;            // "M06.9"
  status: PAStatus;
  submittedDate: string;        // "Apr 4, 2025"
  dueDate: string;              // "Apr 19, 2025"
  payer: string;                // "BlueCross Health"
  payerContact: string;
  provider: string;             // "Dr. Sarah Mitchell"
  npi: string;                  // "1234567890"
  authNumber?: string;          // "BCH-2025-8841" (only if approved)
  authValidFrom?: string;
  authValidUntil?: string;      // "Oct 4, 2025"
  denialReason?: string;
  priority: Priority;
  documents: string[];          // filenames
  submissionChannel: SubmissionChannel;
  previousTreatments?: string;
  clinicalJustification?: string;
  dosage?: string;
  timeline: TimelineEvent[];
  createdAt: string;
  updatedAt: string;
}

export interface TimelineEvent {
  id: string;
  date: string;
  title: string;
  description: string;
  eventType: 'submitted' | 'received' | 'in_review' | 'info_requested' 
           | 'approved' | 'denied' | 'appealed' | 'reauth_started' 
           | 'reauth_approved';
  actor: string;                // "Dr. Sarah Mitchell" or "BlueCross Health"
}

export interface Patient {
  id: string;
  name: string;
  mrn: string;
  dob: string;
  age: number;
  gender: string;
  phone: string;
  email: string;
  insurance: string;
  memberId: string;
  plan: string;
  deductible: string;
  copay: string;
  payer: string;
  payerContact: string;
  conditions: string[];
  medications: PatientMedication[];
  allergies: string[];
  lastVisit: string;
}

export interface PatientMedication {
  name: string;
  dose: string;
  condition: string;
  paStatus: PAStatus;
  authNumber?: string;
  validUntil?: string;
}

export interface ReauthCase {
  id: string;
  patientName: string;
  drug: string;                 // "Humira (adalimumab)"
  drugShort: string;            // "Humira"
  condition: string;
  originalApproval: string;     // "03/15/2023"
  reauthorized: string;         // "06/01/2023"
  newExpiration: string;        // "12/01/2023"
  daysUntilExpiry: number;
  urgency: UrgencyLevel;
  authNumber: string;
}

export interface AppealCase {
  caseId: string;
  patientName: string;
  drug: string;
  denialReason: string;
  deniedDate: string;
  payer: string;
  appealStatus: 'not_filed' | 'filed' | 'upheld' | 'reversed';
  appealLetter?: string;
}

export interface NewPAFormData {
  patientName: string;
  dateOfBirth: string;
  diagnosis: string;
  diagnosisCode: string;
  medication: string;

  previousTreatments: string;

  documents: File[];
  prescribingPhysician: string;
  npi: string;
  payer: string;
  memberId: string;
}

SECTION 9: BACKEND API
════════════════════════════════════════════════════════════════

Express.js server on port 3001. CORS enabled for localhost:5173.
All data stored in memory (module-level arrays, initialized from seed data).

ROUTES:

GET  /api/cases              → returns all cases, supports ?status= and ?payer= filter
GET  /api/cases/:id          → single case with full timeline
POST /api/cases              → create new PA case
  body: NewPAFormData fields
  auto-generates id (PA-XXXXXX), sets status=pending, 
  adds submitted timeline event, returns created case

PATCH /api/cases/:id         → update case
  Allowed fields: status, denialReason, authNumber, authValidFrom, authValidUntil
  Business logic:
    status → 'approved': auto-generate authNumber if not provided
      format: [PAYER_PREFIX]-[YEAR]-[4digits]
      Payer prefixes: BlueCross→BCH, Aetna→ATN, UnitedHealthcare→UHC, 
                      Cigna→CGN, Humana→HUM
    status → 'denied': denialReason required
    Any status change: add timeline event automatically
    All changes update updatedAt timestamp

POST /api/cases/:id/submit   → mark as submitted to payer
  Changes status from 'pending' to 'review'
  Adds timeline event: "Received by Payer" + "Under Clinical Review"
  Returns updated case

POST /api/cases/:id/appeal   → file appeal
  Changes status to 'review' (re-review after denial)
  Adds timeline event: "Appeal Filed"
  Returns updated case

GET  /api/patients           → all patients
GET  /api/patients/:id       → single patient

GET  /api/reauth             → all reauth cases

POST /api/ai                 → AI response
  body: { role: 'provider'|'patient'|'payer', feature: string, message: string }
  Returns mock response from response map based on feature or keyword matching
  Response format: { response: string, feature: string, timestamp: string }

SECTION 10: BUSINESS PROCESS ACCURACY
════════════════════════════════════════════════════════════════

These business rules must be enforced in the application:

1. PA LIFECYCLE (enforced in UI and API):
   new → pending → review → approved | denied
   denied → appealed → review → approved | denied
   approved → reauth_due (when approaching expiry) → reauth_review → reauth_approved

2. PROVIDER CAN:
   - Create new PA (POST /api/cases)
   - View all their cases
   - Attach documents to pending/review cases
   - Share case with payer (submit channel)
   - File appeal for denied cases
   - Start re-auth process for expiring approvals
   Provider CANNOT: approve, deny, or make coverage decisions

3. PATIENT CAN:
   - View their own cases only (filter by patientId)
   - See current journey step in plain language
   - Upload documents (supporting evidence)
   - Track document status
   - Use AI for plain-language explanations
   Patient CANNOT: submit PAs (doctor does that), approve, deny, access other patients

4. PAYER CAN:
   - See all cases in their queue
   - Approve cases (generates auth number)
   - Deny cases (must provide reason)
   - Request additional information (sets status to 'review' with info_requested event)
   Payer CANNOT: create PA requests, file appeals

5. AUTH NUMBER GENERATION:
   Format: [PAYER_PREFIX]-[YYYY]-[4-digit random]
   Valid period: 6 months from approval date (default)
   Auth number stored on case, surfaced to provider and patient immediately

6. DENIAL FLOW:
   Payer denies → denial reason required (not optional)
   Provider notified (shown in dashboard alerts)
   Patient notified (shown in patient dashboard)
   "File Appeal" action enabled for 60 days from denial
   Appeal changes status back to 'review' with 'appealed' timeline event

7. REAUTHORIZATION TRIGGER:
   Cases with validUntil within 45 days → appear in "Upcoming Reauthorizations"
   Cases with validUntil in past → "Recently Reauthorized" table (with newExpiration column)
   Re-auth starts new PA process for same drug/patient

8. DOCUMENT REQUIREMENTS (enforced with visual indicators):
   Required docs (red label): Clinical Notes, Lab Results, Prior Auth Form
   Recommended (amber label): Specialist Letter, Step Therapy Records
   Optional (gray): Imaging/Scans
   AI suggests missing documents based on drug type and payer rules

9. SUBMISSION CHANNELS (business accurate):
   HL7 FHIR API: electronic, <1 hour confirmation, preferred for connected EHRs
   Payer Web Portal: manual, 4-8 hours, good for non-FHIR payers
   Secure Fax: legacy, 24-48 hours, required for some smaller payers

10. PROCESSING TIMELINES (enforced in display):
    Standard PA: 3-5 business days (show on case)
    Urgent/Expedited: 24-72 hours (if priority=high)
    Real-time: <1 hour (FHIR channel)
    Display "Due: [date]" in red if high priority, gray if normal
